# V4.X 프롬프트

# 1. 프롬프트 입력

## V4.0

Few-shot 기반 출력 방식을 유지하면서 **단계적 분석 절차(Chain of Thought, CoT)**를 명시적으로 추가하였다. 회의 목적 파악, 결정사항과 논의사항 구분, 담당자·작업·기한 추출, 리스크 식별 등의 분석 과정을 정의하여 문맥 유지와 출력 일관성을 향상시키고, 내부 추론은 출력하지 않도록 하여 최종 결과만 제공하도록 개선하였다.

```
[V3]

[분석 절차]

1. 회의 목적 파악
2. 결정사항과 논의사항 구분
3. 담당자, 작업, 기한 추출
4. 리스크 식별
5. 정보 부족 시 최대 3개의 확인 질문 생성
6. 내부 추론은 출력하지 않고 최종 결과만 출력
```

## V4.1

결정사항과 논의사항을 구분하는 기준을 명확하게 정의하여, 제안이나 검토 단계의 내용을 결정사항으로 잘못 분류하지 않도록 개선하였다.

```
[V4.0]

[결정사항 판단 기준]

다음 조건을 모두 만족하는 경우에만 결정사항으로 작성합니다.

- 회의에서 명확하게 확정된 내용
- 참석자의 합의 또는 승인된 내용
- "하기로 했다", "확정했다", "진행한다" 등 확정 표현이 있는 내용

다음 내용은 결정사항으로 작성하지 않습니다.

- 제안
- 의견
- 검토 예정 사항
- 가능성
- 예상
- 조건부 내용

위 내용은 "확인 필요" 또는 "리스크"로 분류합니다.
```

## V4.2

회의록에 없는 날짜를 생성하거나 상대적인 날짜를 실제 날짜로 변환하지 않도록 날짜 처리 규칙을 추가하였다.

```
[V4.1]

[날짜 처리 규칙]

- 회의록에 없는 날짜를 생성하지 않습니다.
- 상대적인 날짜 표현(다음 주, 이번 주 금요일 등)은 원문 그대로 유지합니다.
- 실제 날짜로 계산하거나 변환하지 않습니다.
```

## V4.3

최종 결과 출력 전에 Hallucination 여부를 검증하는 절차를 추가하여, 사실 기반 출력의 신뢰성을 강화하였다.

```
[V4.2]

[Hallucination 검증]

최종 결과를 출력하기 전에 다음 사항을 확인합니다.

- 회의록에 없는 내용을 생성하지 않았는가?
- 제안 사항을 결정사항으로 작성하지 않았는가?
- 회의록에 없는 담당자나 기한을 생성하지 않았는가?
- 정보가 부족한 내용은 "확인 필요"로 분류하였는가?

검증 결과 문제가 발견되면 수정 후 최종 결과를 출력합니다.
```

# 3. V4 프롬프트 분석

## 1. 프롬프트 입력

V4 프롬프트는 V3의 Few-shot 기반 출력 방식을 유지하면서 단계적 분석 절차(CoT), 결정사항 판단 기준, 날짜 처리 규칙, Hallucination 검증 절차를 순차적으로 추가하여 사실 기반 문서 생성의 신뢰성을 높이는 것을 목표로 하였다.

V4.0에서는 회의록을 일정한 순서로 분석하는 절차를 정의하여 문맥 유지와 출력 일관성을 향상시켰다.

V4.1에서는 결정사항과 논의사항을 구분하는 기준을 명확하게 정의하여 제안이나 검토 단계의 내용을 결정사항으로 잘못 분류하는 문제를 개선하였다.

V4.2에서는 회의록에 없는 날짜를 생성하거나 상대적인 날짜를 실제 날짜로 변환하지 않도록 날짜 처리 규칙을 추가하였다.

V4.3에서는 최종 결과 출력 전에 Hallucination 여부를 자체 검증하는 절차를 추가하여 사실 기반 출력의 신뢰성을 더욱 강화하였다.

---

## 2. LLM별 출력 결과 비교

동일한 V4 프롬프트를 GPT-5.5, Claude Sonnet 4.6, Gemini 3.1 Pro에 적용하여 결과를 비교하였다.

### GPT-5.5

GPT-5.5는 V4에서 추가된 결정사항 판단 기준을 가장 적극적으로 반영하였다. 이전 버전에서 결정사항으로 작성하던 일부 항목을 확인 필요로 이동하여 사실 기반 출력이 더욱 강화되었다. 또한 날짜 변환 규칙을 준수하여 상대적인 날짜 표현을 그대로 유지하였다.

다만 결정사항을 보수적으로 판단하는 경향이 있어 일부 실제 합의된 내용까지 확인 필요 항목으로 분류하는 사례가 확인되었다.

### Claude Sonnet 4.6

Claude Sonnet 4.6은 리스크를 일정, 서버, 비용, 보안 등으로 체계적으로 분류하였으며, 프로젝트 관리 문서 형태를 가장 안정적으로 유지하였다. V4에서 추가된 날짜 처리 규칙을 반영하여 이전 버전에서 나타났던 실제 날짜 변환이 제거되었으며, Hallucination도 감소하였다.

그러나 일부 제안 사항을 결정사항으로 판단하는 경향은 여전히 일부 확인되었다.

### Gemini 3.1 Pro

Gemini 3.1 Pro는 V3보다 출력 형식이 더욱 안정적으로 유지되었으며, 확인 필요 항목도 보다 명확하게 작성하였다. 날짜 처리 규칙도 정상적으로 반영하여 회의록의 상대적인 날짜 표현을 그대로 유지하였다.

다만 Action Items와 리스크는 여전히 다른 모델보다 간략하게 작성되는 경향이 있었으며, 일부 세부 업무가 누락되는 경우가 확인되었다.

## 3. V4 프롬프트의 개선 효과

V4 프롬프트는 단계적 분석 절차와 사실 기반 규칙을 강화함으로써 다음과 같은 개선 효과를 확인하였다.

### ① 결정사항 분류 정확도 향상

결정사항 판단 기준을 추가함으로써 제안, 의견, 검토 예정 사항을 결정사항으로 잘못 분류하는 사례가 감소하였다.

### ② 날짜 처리 일관성 향상

상대적인 날짜를 실제 날짜로 변환하지 않도록 규칙을 추가하여 회의록 원문의 정보를 그대로 유지할 수 있었다.

### ③ Hallucination 감소

최종 출력 전에 자체 검증 절차를 수행하도록 하여 회의록에 없는 담당자, 일정, 작업을 생성하는 사례가 감소하였다.

### ④ 문맥 유지 능력 향상

단계적 분석 절차(CoT)를 통해 장문의 회의록에서도 핵심 정보를 안정적으로 유지하며 프로젝트 관리 문서를 생성하였다.

### ⑤ 출력 신뢰성 향상

결정사항, Action Items, 리스크, 확인 필요 항목이 더욱 명확하게 구분되어 실무 활용성이 향상되었다.

---

## 4. V4 프롬프트의 한계

V4 프롬프트에서도 다음과 같은 한계가 확인되었다.

### ① 결정사항 판단 기준의 차이

동일한 규칙을 적용하더라도 모델마다 결정사항과 확인 필요를 구분하는 기준에는 차이가 존재하였다.

### ② 모델별 출력 성향

GPT-5.5는 사실 기반을 우선하여 보수적으로 판단하는 경향을 보였으며, Claude Sonnet 4.6은 구조화된 문서를 생성하는 데 강점을 보였다. Gemini 3.1 Pro는 간결성을 유지하는 대신 세부 정보가 일부 생략되는 경향이 있었다.

### ③ 정보량 차이

동일한 프롬프트에서도 Action Items와 리스크의 상세 수준은 모델마다 차이가 있었으며, 일부 업무 항목은 누락되기도 하였다.

### ④ Hallucination 완전 제거의 어려움

Hallucination 검증 절차를 추가하였음에도 불구하고, 일부 모델에서는 회의 내용을 해석하는 과정에서 경미한 추론이 포함되는 사례가 확인되었다.

---

## 5. 개선 필요 사항

V4 실험 결과를 바탕으로 다음 사항을 추가 개선할 필요가 있다고 판단하였다.

- 다양한 회의록 유형을 추가하여 Few-shot 예시를 확대한다.
- 장문의 회의록에 대한 문맥 유지 성능을 지속적으로 검증한다.
- Hallucination 검증 기준을 더욱 세분화하여 사실 기반 출력을 강화한다.
- 결정사항과 확인 필요를 구분하는 기준을 다양한 사례로 보완한다.
- 프로젝트 유형별(개발, 운영, 기획 등) 맞춤형 예시를 추가하여 일반화 성능을 향상시킨다.

---

## 6. 분석 결과

V4 프롬프트는 CoT 기반 분석 절차와 결정사항 판단 기준, 날짜 처리 규칙, Hallucination 검증 절차를 추가함으로써 지금까지의 버전 중 가장 안정적인 결과를 생성하였다. 특히 회의록 원문에 근거한 사실 기반 출력과 문맥 유지 능력이 향상되었으며, 프로젝트 관리 문서의 구조도 더욱 일관되게 유지되었다.

그러나 모델마다 결정사항을 판단하는 기준과 정보 표현 방식에는 여전히 차이가 존재하였다. 또한 장문의 회의록이나 다양한 프로젝트 유형에 대해서는 추가적인 검증이 필요하였다.

전체적으로 V4 프롬프트는 실무 환경에서 프로젝트 관리 문서를 자동 생성하기에 가장 높은 완성도를 보였으며, Hallucination을 최소화하면서도 일관된 출력 형식을 제공하는 최종 프롬프트로 활용할 수 있음을 확인하였다.
