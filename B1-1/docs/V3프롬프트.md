# 3. V3 프롬프트 분석

## 1. 프롬프트 입력

V3 프롬프트는 V2에서 구축한 역할 정의, 출력 형식, 사실 기반 작성 규칙을 유지하면서 **Few-shot 예시를 단계적으로 추가**하여 모델의 출력 일관성과 정확성을 향상시키는 것을 목표로 하였다.

V3.0에서는 정상 입력 예시 1개를 추가하여 원하는 출력 형식을 학습하도록 구성하였다.

V3.1에서는 정상 입력 예시를 2개로 확대하고, 모호한 입력 예시를 추가하여 정보가 부족한 경우 확인 질문을 먼저 수행하도록 개선하였다.

---

## 2. LLM별 출력 결과 비교

동일한 V3 프롬프트를 GPT-5.5, Claude Sonnet 4.6, Gemini 3.1 Pro에 적용하여 결과를 비교하였다.

### GPT-5.5

GPT-5.5는 회의 목적, 결정사항, Action Items, 리스크, 확인 필요 항목을 가장 균형 있게 정리하였다. Few-shot 예시를 반영하여 출력 형식을 안정적으로 유지하였으며, 담당자와 기한도 비교적 정확하게 추출하였다.

다만 일부 제안 사항을 결정사항으로 해석하거나, 회의 내용을 기반으로 Action Items를 적극적으로 생성하는 경향은 여전히 일부 확인되었다.

### Claude Sonnet 4.6

Claude Sonnet 4.6은 가장 상세하고 구조적인 결과를 생성하였다. 리스크를 일정, 서버, 비용, 보안, 품질 등으로 체계적으로 분류하여 실무 보고서 형태에 가장 가까운 결과를 제공하였다.

반면 회의록에 없는 날짜를 실제 일정으로 변환하거나 일부 내용을 확정된 사실처럼 표현하는 경우가 있었으며, 정보량이 많아 문서가 다소 길어지는 경향이 나타났다.

### Gemini 3.1 Pro

Gemini 3.1 Pro는 출력 형식이 V2보다 안정적으로 유지되었으며, Few-shot 예시를 반영하여 불필요한 안내 문구가 크게 감소하였다. 핵심 내용을 간결하게 정리하여 가독성이 향상되었다.

그러나 Action Items와 리스크 항목은 다른 모델보다 간략하게 작성되는 경향이 있었으며, 일부 세부 정보가 누락되는 경우가 확인되었다.

---

## 3. V3 프롬프트의 개선 효과

V3 프롬프트는 Few-shot 예시를 적용함으로써 다음과 같은 개선 효과를 확인하였다.

### ① 출력 형식의 일관성 향상

예시를 통해 원하는 출력 구조를 제시함으로써 모델 간 출력 형식 차이가 감소하였다.

### ② 결정사항 구분 능력 향상

정상 입력 예시를 제공함으로써 회의 목적, 결정사항, Action Items를 보다 명확하게 구분하는 경향이 나타났다.

### ③ 모호한 입력 처리 개선

모호한 입력 예시를 추가하여 정보가 부족한 경우 임의로 내용을 생성하기보다 확인 질문을 우선 수행하는 비율이 증가하였다.

### ④ 출력 품질 향상

Markdown 형식과 표 구조를 안정적으로 유지하였으며, 실무에서 바로 활용 가능한 프로젝트 관리 문서 형태가 더욱 일관되게 생성되었다.

### ⑤ Hallucination 감소

Few-shot 예시를 통해 회의록에 없는 내용을 생성하는 빈도가 감소하였으며, 확정되지 않은 내용은 확인 필요로 분류하는 경향이 강화되었다.

---

## 4. V3 프롬프트의 한계

V3 프롬프트에서도 다음과 같은 한계가 확인되었다.

### ① 결정사항 판단 기준 차이

동일한 Few-shot을 제공하더라도 일부 모델은 제안이나 의견을 결정사항으로 해석하는 경우가 있었다.

### ② 모델별 출력 성향 차이

GPT-5.5는 적극적으로 정보를 구조화하는 경향이 있었고, Claude Sonnet 4.6은 상세한 설명을 제공하였으며, Gemini 3.1 Pro는 간결성을 우선하는 경향을 보였다.

### ③ 정보량 차이

Action Items와 리스크의 세부 수준은 모델마다 차이가 있었으며, 동일한 프롬프트를 사용해도 출력 정보량이 완전히 동일하지는 않았다.

### ④ 예시 의존성

Few-shot 예시와 유사한 형태의 회의록에서는 높은 성능을 보였지만, 예시와 다른 유형의 입력에서는 출력 품질이 일부 달라질 가능성이 확인되었다.

---

## 5. 개선 필요 사항

V3 실험 결과를 바탕으로 다음 사항을 추가 개선할 필요가 있다고 판단하였다.

- 다양한 유형의 회의록 예시를 추가하여 일반화 성능을 높인다.
- 결정사항과 논의사항을 구분하는 기준을 더욱 명확하게 정의한다.
- Hallucination 방지를 위한 사실 기반 규칙을 강화한다.
- 확인 질문이 필요한 상황과 확인 필요 항목으로 처리할 상황을 명확하게 구분한다.
- 리스크와 Action Items 작성 기준을 더욱 세분화하여 출력 일관성을 향상시킨다.

---

## 6. 분석 결과

V3 프롬프트는 Few-shot 예시를 단계적으로 추가함으로써 출력 형식의 일관성과 사실 기반 문서 작성 능력을 크게 향상시켰다. 특히 정상 입력과 모호한 입력 예시를 함께 제공함으로써 정보가 부족한 경우의 처리 방식이 더욱 안정적으로 개선되었다.

그러나 모델마다 결정사항 해석 기준과 출력 정보량에는 여전히 차이가 존재하였다. 또한 예시와 유사한 입력에서는 높은 성능을 보였지만, 다양한 형태의 회의록에 대한 일반화 성능을 높이기 위해서는 추가적인 예시와 검증 과정이 필요하였다.

따라서 다음 단계(V4)에서는 Few-shot 예시를 유지하면서도 단계적 분석 절차를 더욱 명확히 정의하고, Hallucination 검증과 문맥 유지 능력을 강화하여 보다 안정적인 프로젝트 관리 문서 생성 프롬프트로 발전시키고자 하였다.
