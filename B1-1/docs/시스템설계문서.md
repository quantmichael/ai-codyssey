## 1. 문제 정의

### 업무 과업

회의록을 분석하여 프로젝트 관리 문서를 자동 생성한다.

생성 결과에는 다음 항목을 포함한다.

- 회의 목적
- 결정사항
- Action Items
- 리스크
- 확인 필요 사항

회의록에 없는 내용은 추측하지 않으며, 사실에 근거한 결과만 출력한다.

---

## 2. 타겟 사용자

- 프로젝트 매니저(PM)
- 기획자
- 개발자
- QA 담당자
- 프로젝트 참여자

회의 후 회의록을 빠르게 프로젝트 관리 문서로 변환해야 하는 실무자를 대상으로 한다.

---

## 3. 업무 과업 입력 템플릿

```
[업무 과업]

회의록 요약 및 프로젝트 관리 문서 생성

[입력]

회의록 원문

[출력]

- 회의 목적
- 결정사항
- Action Items
- 리스크
- 확인 필요

[톤]

실무 보고서 형식

[금지]

- 회의록에 없는 내용 추측 금지
- 확정되지 않은 내용을 결정사항으로 작성 금지

[확인 질문]

정보가 부족하면 먼저 필요한 내용을 질문한다.
```

---

## 4. 입력 데이터 형태

본 시스템은 프로젝트 회의록을 입력으로 사용한다.

입력 데이터에는 다음 정보가 포함될 수 있다.

- 프로젝트명
- 회의 일시
- 참석자
- 회의 내용
- 추가 요청사항(선택)

입력 형식은 자유 텍스트이며, 회의록에 포함된 내용을 기준으로 프로젝트 관리 문서를 생성한다.

---

## 5. 출력 규격

출력 결과는 Markdown 기반의 프로젝트 관리 문서 형식으로 작성한다.

출력 항목은 다음과 같다.

```
## 회의 목적

## 결정사항

## Action Items

| 담당자 | 작업 | 기한 |

## 리스크

## 확인 필요
```

출력 시 다음 규칙을 적용한다.

- 회의록에 없는 내용은 추측하지 않는다.
- 확정되지 않은 내용은 결정사항으로 작성하지 않는다.
- 담당자 또는 기한이 명확하지 않으면 "확인 필요"에 작성한다.
- 실무자가 바로 사용할 수 있는 문서 형태로 작성한다.

## 7. System Prompt

```
당신은 IT 기업에서 10년 이상 프로젝트를 수행한 시니어 프로젝트 매니저(Project Manager)입니다.

회의록을 분석하여 프로젝트 관리 문서를 작성하는 전문가입니다.

다음 규칙을 반드시 따르세요.

[목표]

회의록을 분석하여

- 회의 목적
- 결정사항
- Action Items
- 리스크
- 확인 필요 사항

을 추출합니다.

[규칙]

- 회의록에 없는 내용을 추측하지 않습니다.
- 확정되지 않은 내용은 결정사항으로 작성하지 않습니다.
- 담당자 또는 기한이 명확하지 않으면 "확인 필요"에 작성합니다.
- 정보가 부족하면 먼저 필요한 내용을 질문합니다.
- 내부 추론은 출력하지 않고 최종 결과만 출력합니다.
- 실무자가 바로 사용할 수 있는 프로젝트 관리 문서 형식으로 작성합니다.
```

## 10. Hallucination 검증 설계

회의록의 사실을 기반으로 5개의 질문을 생성하였다.

| 질문 | 기대 답변 |
| --- | --- |
| API 문서는 언제 공유하기로 했나요? | 이번 주 금요일 |
| 로그인 기능은 어떻게 처리하기로 했나요? | 보안 검토 필요 |
| PDF 문서 검색 기능은 언제 검토하기로 했나요? | 2차 개발 |
| 테스트 기간은 얼마나 필요한가요? | 최소 2주 |
| 서버는 무엇으로 확정되었나요? | 아직 확정되지 않음 |

### Pass 기준

- 정답을 말하고 회의록 근거를 제시하거나 확인 가능한 방법을 설명한다.
- 정보가 부족하면 "확인 필요"를 명시한다.

### Fail 기준

- 회의록에 없는 사실을 생성한다.
- 미확정 내용을 확정된 사실처럼 답변한다.

## 11. 프롬프트 개선 이력

### V1

**간단한 회의록 요약 프롬프트**

**문제점**

- 출력 형식이 일정하지 않음
- 결정사항과 논의사항이 혼재됨

---

### V2

**프로젝트 관리 문서 형식 추가**

**개선사항**

- 출력 템플릿 지정
- 추측 금지 규칙 추가

---

### V3

**Few-shot 적용**

**개선사항**

- 정상 입력 예시 2개
- 모호한 입력 예시 1개
- 출력 일관성 향상

---

### V4

**고도화된 Chain of Thought(CoT) 적용**

**개선사항**

분석 절차를 추가하였다.

1. 회의 목적 파악
2. 결정사항과 논의사항 구분
3. 담당자·작업·기한 추출
4. 리스크 식별
5. 정보 부족 시 확인 질문 생성
6. 내부 추론은 출력하지 않고 최종 결과만 출력

추가적으로 다음 규칙을 적용하였다.

- 결정사항 판단 기준 명확화
- 회의록에 없는 날짜 생성 및 날짜 변환 금지
- Hallucination 방지 규칙 및 검증 절차 추가
- 사실 기반 출력 강화

조건 변경과 추가 정보를 반영하는 문맥 유지 성능이 향상되었으며, 회의록에 없는 내용을 생성하지 않도록 개선하였다.
