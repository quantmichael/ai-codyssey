아래 형식으로 회의록을 요약하세요.

## 결정사항

## Action Items

## **V2.1**

회의록에 없는 내용을 생성하지 않도록 **사실 기반 작성 규칙**을 추가하고, 확정되지 않은 내용은 **"확인 필요"** 항목으로 분리하도록 개선하였다.

```
[V2.0]

규칙:

- 회의록에 명확히 나온 내용만 작성하세요.
- 확정되지 않은 내용은 결정사항으로 쓰지 말고 "확인 필요" 항목에 작성하세요.
```

## **V2.2**

회의 목적 파악, 결정사항 구분, 리스크 식별 등 **기본적인 분석 절차(CoT)**를 추가하여 결과의 정확성과 일관성을 향상하였다.

```
[V2.1]

회의록을 바로 요약하지 말고,

1. 회의 목적을 파악합니다.

2. 결정된 사항과 논의 중인 사항을 구분합니다.

4. 리스크와 미확정 사항을 식별합니다.

5. 최종 결과를 지정된 형식으로 출력합니다.

내부 검토 과정은 출력하지 말고,

최종 결과만 작성하세요.
```

## **V2.3**

정보가 부족한 경우 **확인 질문을 먼저 생성하는 절차**를 추가하여 불확실한 정보를 임의로 생성하지 않도록 개선하였다.

```
[V2.2]

회의록에 필요한 정보가 부족하면

최대 3개의 확인 질문을 먼저 하세요.

충분한 정보가 있으면

질문 없이 바로 요약을 작성하세요.
```

## **V2.4**

최종 결과를 **실무자가 바로 활용할 수 있는 수준의 간결하고 명확한 문서**로 작성하도록 출력 품질 기준을 강화하였다.

```
[V2.3]

최종 결과는

실무자가 바로 사용할 수 있는 수준으로

간결하고 명확하게 작성하세요.
```

## **V2** Final - System Prompt

역할(Role), 목적(Objective), 분석 절차(Workflow), 출력 형식(Output Format), 제약 조건(Constraints), 예외 처리(Failure Handling), 품질 기준(Quality Criteria)을 통합하여 **완전한 시스템 프롬프트**를 구성하였다.

**System Prompt**는 모델의 역할, 목표, 규칙, 출력 형식처럼 매번 변하지 않는 기본 동작 원칙을 정의하기 위해 분리하였다.

회의록을 분석하여 프로젝트 관리 문서를 작성하는 전문가이며,

일부 정보는 누락되거나 불명확할 수 있습니다.

3. 담당자와 Action Items를 추출합니다.

5. 정보가 부족한 부분을 확인합니다.

6. 지정된 출력 형식으로 결과를 작성합니다.

# 3. V2 프롬프트 분석

## 3.1. 프롬프트 입력

V2 프롬프트는 V1에서 확인된 출력 형식 불일치, 추론 포함, 불필요한 설명 출력 문제를 개선하기 위해 단계적으로 보완하였다.

V2.1에서는 시니어 프로젝트 매니저 역할과 기본 출력 형식을 지정하였다.

V2.2에서는 회의록에 없는 내용을 작성하지 않도록 사실 기반 규칙을 추가하였다.

V2.3에서는 회의 목적 파악, 결정사항 구분, 리스크 식별 등 기본적인 분석 절차를 추가하였다.

V2.4에서는 정보 부족 시 확인 질문을 하도록 예외 처리 방식을 추가하였다.

V2.5에서는 최종 결과가 실무자가 바로 사용할 수 있도록 간결성과 명확성을 강화하였다.

최종적으로 V2 Final에서는 역할, 목표, 업무 절차, 출력 형식, 규칙을 하나의 프롬프트로 통합하여 실제 업무 환경에서 바로 사용할 수 있는 형태로 구성하였다.

---

## 3.2. LLM별 출력 결과 비교

동일한 V2 프롬프트를 GPT-5.5, Claude Sonnet 4.6, Gemini 3.1 Pro에 적용하여 결과를 비교하였다.

### GPT-5.5

GPT-5.5는 회의 목적, 결정사항, Action Items, 리스크, 확인 필요 항목을 비교적 균형 있게 정리하였다. 출력 형식도 지정한 구조를 잘 따랐으며, 담당자와 기한을 표 형태로 정리하여 실무 활용성이 높았다.

다만 일부 항목에서는 회의록의 발언을 바탕으로 작업 담당자나 세부 처리 방식을 적극적으로 정리하는 경향이 있었다. 따라서 사실 기반 문서 작성 관점에서는 일부 항목을 확인 필요로 분리할 필요가 있었다.

### Claude Sonnet 4.6

Claude Sonnet 4.6은 가장 구조적이고 상세한 결과를 생성하였다. 결정사항, 리스크, 확인 필요 항목을 체계적으로 구분하였고, 일정 리스크·비용 리스크·보안 리스크·품질 리스크처럼 리스크를 유형별로 정리한 점이 장점이었다.

하지만 일부 날짜를 구체적으로 변환하거나, 회의록에 명시되지 않은 표현을 확정된 일정처럼 작성한 부분이 있었다. 또한 설명이 다소 길어질 수 있어 간결한 실무 문서가 필요한 경우에는 조정이 필요하였다.

### Gemini 3.1 Pro

Gemini 3.1 Pro는 핵심 내용을 간결하고 이해하기 쉽게 정리하였다. 전체 구조는 지정한 출력 형식을 따랐으며, 주요 결정사항과 확인 필요 항목을 빠르게 파악할 수 있었다.

다만 Action Items가 일부만 포함되어 정보량이 부족하였고, 리스크도 비교적 간략하게 작성되었다. 실무 자동화 문서로 사용하기에는 일부 세부 항목이 누락될 가능성이 있었다.

## 3.3. V2 프롬프트의 개선 효과

V2 프롬프트는 V1에 비해 다음과 같은 개선 효과를 보였다.

### ① 출력 형식의 일관성 향상

V1에서는 모델마다 결과 구조가 달랐으나, V2에서는 회의 목적, 결정사항, Action Items, 리스크, 확인 필요 항목으로 출력 형식이 일정하게 유지되었다.

### ② Action Items 구조화 개선

담당자, 작업, 기한을 표로 작성하도록 지정하여 후속 업무를 실무자가 바로 확인할 수 있게 되었다.

### ③ 사실 기반 작성 규칙 강화

회의록에 없는 내용을 추측하지 말라는 규칙과 확정되지 않은 내용을 확인 필요로 분리하는 규칙을 추가하여 Hallucination 발생 가능성을 줄였다.

### ④ 리스크 식별 능력 향상

단순 요약이 아니라 리스크를 별도 항목으로 추출하도록 하여 일정, 보안, 비용, 품질 관련 위험 요소를 더 명확하게 파악할 수 있었다.

### ⑤ 실무 활용성 향상

Markdown 형식과 표 구조를 적용하여 회의 후 바로 공유 가능한 프로젝트 관리 문서 형태에 가까워졌다.

## 3.4. V2 프롬프트의 한계

V2 프롬프트는 V1보다 개선되었지만 다음과 같은 한계도 확인되었다.

### ① 확정 여부 판단의 어려움

회의록에서 제안된 내용과 실제로 확정된 내용을 모델이 완전히 구분하지 못하는 경우가 있었다. 예를 들어 UI 배치, 브랜드 컬러 적용, FAQ 기반 자동 응답 등은 회의 중 언급되었지만 명시적 승인 여부가 불분명할 수 있다.

### ② 일부 모델의 과도한 구체화

Claude Sonnet 4.6은 다음 회의 일정을 실제 날짜로 변환하는 등 원문보다 구체적인 표현을 생성하였다. 이는 편리할 수 있지만, 사실 기반 문서에서는 원문에 없는 정보로 판단될 수 있다.

### ③ 모델별 정보량 차이

GPT-5.5와 Claude Sonnet 4.6은 비교적 상세한 결과를 생성한 반면, Gemini 3.1 Pro는 일부 Action Items와 리스크를 간략하게 작성하였다. 동일한 프롬프트를 사용해도 모델별 출력 밀도 차이는 여전히 존재하였다.

### ④ 확인 필요 항목의 기준 차이

각 모델이 어떤 내용을 확인 필요로 분류할지에 대한 기준이 완전히 동일하지 않았다. 따라서 더 엄격한 기준을 적용하려면 Few-shot 예시가 추가로 필요하다.

---

## 3.5. 개선 필요 사항

V2 실험 결과를 바탕으로 다음 사항을 추가 개선할 필요가 있다고 판단하였다.

- 결정사항과 논의사항을 더 엄격하게 구분하도록 한다.
- 명시적 승인 발언이 없는 항목은 결정사항이 아니라 확인 필요로 분류하도록 한다.
- 회의록에 없는 날짜 변환이나 일정 계산을 하지 않도록 규칙을 추가한다.
- 담당자 또는 기한이 불명확한 경우 Action Items에 넣더라도 확인 필요 표시를 명확히 한다.
- 정상 입력 예시와 모호한 입력 예시를 추가하여 모델의 출력 기준을 더 안정화한다.
- 리스크를 일정, 보안, 비용 등 유형별로 분류하는 예시를 제공한다.

---

## 3.6. 분석 결과

V2 프롬프트는 V1보다 출력 형식, 사실 기반 규칙, 실무 활용성 측면에서 뚜렷하게 개선되었다. 특히 회의 목적, 결정사항, Action Items, 리스크, 확인 필요 항목을 고정된 구조로 출력하도록 한 점은 결과의 일관성을 높이는 데 효과적이었다.

그러나 제안된 내용과 확정된 내용을 구분하는 기준은 여전히 일부 불안정하였다. 또한 모델에 따라 세부 정보의 포함 범위와 확인 필요 항목 분류 방식이 달랐다.

따라서 다음 단계에서는 Few-shot 예시를 추가하여 정상 입력, 모호한 입력, 확정 여부가 불분명한 입력에 대한 처리 기준을 더 명확히 제시할 필요가 있다. 이를 통해 모델이 회의록에 근거한 사실만 출력하고, 불확실한 내용은 확인 필요로 분리하도록 개선할 수 있다.
