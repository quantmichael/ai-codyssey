# 3. V1 프롬프트 분석

## 3.1. 사용한 V1 프롬프트

```
당신은 프로젝트 매니저입니다.

아래 회의록을 읽고 요약해 주세요.

회의록:

(회의록 원문)
```

본 프롬프트는 역할(Role)만 지정하고, 출력 형식이나 세부 규칙은 제공하지 않은 간단한 형태이다. 이를 통해 각 LLM이 동일한 입력에 대해 어떤 방식으로 정보를 이해하고 구조화하는지 비교하고자 하였다.

---

## 3.2. 모델별 결과 비교

동일한 회의록을 GPT-5.5, Claude Sonnet 4.6, Gemini 3.1 Pro에 입력하여 결과를 비교하였다.

### GPT-5.5

**장점**

- 프로젝트 목적, 주요 논의, 결정사항, Action Items, 리스크, 다음 회의까지 가장 체계적으로 구분하였다.
- 회의 내용을 빠짐없이 반영하였으며, 실무에서 바로 활용 가능한 수준의 결과를 생성하였다.

**단점**

- 일부 Action Items는 회의 내용을 바탕으로 약간의 추론이 포함되었다.

---

### Claude Sonnet 4.6

**장점**

- 표 형식이 깔끔하고 핵심 내용 위주로 간결하게 정리하였다.
- 회의의 핵심을 빠르게 파악하기 쉬운 결과를 생성하였다.

**단점**

- 일부 세부 내용과 리스크가 생략되어 정보량은 GPT-5.5보다 적었다.

---

### Gemini 3.1 Pro

**장점**

- 설명이 친절하고 읽기 쉬운 형태로 작성하였다.
- 초보자도 이해하기 쉬운 문체를 사용하였다.

**단점**

- 업무 자동화 관점에서는 불필요한 안내 문구와 부가 설명이 포함되었다.
- 일부 표현은 원문보다 확장되어 작성되었다.

---

## 3.3. V1 프롬프트의 한계

실험 결과 다음과 같은 한계를 확인하였다.

### ① 출력 형식의 불일치

동일한 프롬프트를 사용하였음에도 모델마다 서로 다른 구조로 결과를 생성하였다. 일부 모델은 표를 사용하였고, 일부는 문장 중심으로 작성하여 결과 형식의 일관성이 부족하였다.

### ② Action Items 표현 방식 차이

모델마다 담당자와 작업을 정리하는 방식이 달랐다. 일부는 상세하게 작성하였고, 일부는 핵심 내용만 요약하였다. 업무 자동화에서는 일정한 출력 형식이 필요하므로 개선이 요구되었다.

### ③ 불필요한 설명 포함

Gemini는 인사말, 학습 안내, 튜터의 팁 등 업무 결과와 직접 관련 없는 내용을 함께 출력하였다. 이러한 내용은 자동화 결과물에서는 제거될 필요가 있었다.

### ④ 추론 내용 포함

GPT-5.5와 Claude Sonnet 4.6은 회의 내용을 기반으로 일부 작업이나 표현을 보완하여 작성하였다. 이는 실무에서는 도움이 될 수 있지만, 사실 기반 문서 작성에서는 원문에 없는 내용을 생성할 가능성이 있었다.

### ⑤ 출력 기준 부족

출력 형식을 지정하지 않았기 때문에 모델마다 강조하는 내용이 달랐다. 일부는 리스크를 중심으로, 일부는 일정 중심으로 요약하여 결과의 일관성이 부족하였다.

---

## 3.4. 개선 필요 사항

V1 실험 결과를 바탕으로 다음 사항을 개선할 필요가 있다고 판단하였다.

- 출력 형식을 명확하게 지정한다.
- 결정사항, Action Items, 리스크를 일정한 구조로 출력하도록 한다.
- 회의록에 없는 내용은 추측하지 않도록 규칙을 추가한다.
- 정보가 부족한 경우에는 임의로 작성하지 않고 **"확인 필요"**로 표시하도록 한다.
- 실무에서 바로 사용할 수 있도록 Markdown 형식과 표 구조를 적용한다.
- 필요 시 확인 질문을 통해 누락된 정보를 먼저 확인하도록 개선한다.

---

## 3.5. 분석 결과

V1 프롬프트는 최소한의 지시만 제공하였기 때문에 각 LLM의 기본적인 요약 능력을 비교하기에는 적합하였다. 그러나 출력 형식과 정보 표현 방식이 모델마다 크게 달라 결과의 일관성이 부족하였다. 또한 일부 모델은 원문에 없는 내용을 추론하거나 불필요한 설명을 추가하는 경향이 나타났다.
