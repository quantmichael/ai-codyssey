# 1. 비교 목적

동일한 업무 과업(회의록 기반 프로젝트 관리 문서 생성)에 대해 3개의 LLM(GPT-5.5, Claude Sonnet 4.6, Gemini 3.1 Pro)을 비교하였다. 동일한 프롬프트(System Prompt, Few-shot, CoT 포함)를 적용하여 결과의 정확성, 문맥 유지 능력, Hallucination 발생 여부, 실무 활용성을 평가하고 최종 사용할 모델을 선정하였다.

# 7. 모델별 결과 요약

## GPT-5.5

### 장점

- 회의록의 핵심 내용을 정확하게 요약하였다.
- 출력 형식을 일관되게 유지하였다.
- 조건 변경 및 추가 정보를 자연스럽게 반영하였다.
- Hallucination이 확인되지 않았다.

### 단점

- 일부 Action Items를 적극적으로 생성하는 경향이 있었다.

---

## Claude Sonnet 4.6

### 장점

- 가장 구조적인 문서를 생성하였다.
- 회의록의 근거를 함께 제시하였다.
- 리스크를 체계적으로 구분하였다.
- 문맥 유지 능력이 매우 우수하였다.

### 단점

- 설명이 다소 길어질 수 있다.

---

## Gemini 3.1 Pro

### 장점

- 이해하기 쉬운 문장으로 응답하였다.
- 핵심 내용을 간결하게 정리하였다.
- Hallucination이 확인되지 않았다.

### 단점

- 출력 형식이 다소 자유로워 실무 문서 형식과 차이가 있었다.
- 일부 Action Items와 리스크를 간략하게 표현하였다.

# 9. 최종 선정 모델

## GPT-5.5

### 선정 근거

1. 회의록의 핵심 내용을 정확하게 요약하면서도 불필요한 정보를 최소화하였다.
2. Few-shot, CoT, 조건 변경 및 추가 정보를 안정적으로 반영하여 문맥 유지 성능이 우수하였다.
3. 출력 형식이 일관되어 프로젝트 관리 문서 작성과 같은 실무 업무에 바로 활용하기 적합하였다.

Claude Sonnet 4.6 역시 매우 우수한 결과를 제공하였으나, 설명이 다소 길어지는 경향이 있었다. Gemini 3.1 Pro는 이해하기 쉬운 답변을 제공했지만 출력 형식의 일관성과 실무 문서 구성 측면에서는 상대적으로 아쉬운 점이 있었다.

따라서 본 과제의 업무 특성(회의록 기반 프로젝트 관리 문서 작성)에는 **GPT-5.5를 최종 모델로 선정**하였다.

## 1. 문제 정의

### 업무 과업

회의록을 분석하여 프로젝트 관리 문서를 자동 생성한다.

생성 결과에는 다음 항목을 포함한다.

- 회의 목적
- 결정사항
- Action Items
- 리스크
- 확인 필요 사항

회의록에 없는 내용은 추측하지 않으며, 사실에 근거한 결과만 출력한다.

---

## 2. 타겟 사용자

- 프로젝트 매니저(PM)
- 기획자
- 개발자
- QA 담당자
- 프로젝트 참여자

회의 후 회의록을 빠르게 프로젝트 관리 문서로 변환해야 하는 실무자를 대상으로 한다.

---

## 3. 업무 과업 입력 템플릿

```
[업무 과업]

회의록 요약 및 프로젝트 관리 문서 생성

[입력]

회의록 원문

[출력]

- 회의 목적
- 결정사항
- Action Items
- 리스크
- 확인 필요

[톤]

실무 보고서 형식

[금지]

- 회의록에 없는 내용 추측 금지
- 확정되지 않은 내용을 결정사항으로 작성 금지

[확인 질문]

정보가 부족하면 먼저 필요한 내용을 질문한다.
```

---

## 4. 입력 데이터 형태

본 시스템은 프로젝트 회의록을 입력으로 사용한다.

입력 데이터에는 다음 정보가 포함될 수 있다.

- 프로젝트명
- 회의 일시
- 참석자
- 회의 내용
- 추가 요청사항(선택)

입력 형식은 자유 텍스트이며, 회의록에 포함된 내용을 기준으로 프로젝트 관리 문서를 생성한다.

---

## 5. 출력 규격

출력 결과는 Markdown 기반의 프로젝트 관리 문서 형식으로 작성한다.

출력 항목은 다음과 같다.

```
## 회의 목적

## 결정사항

## Action Items

| 담당자 | 작업 | 기한 |

## 리스크

## 확인 필요
```

출력 시 다음 규칙을 적용한다.

- 회의록에 없는 내용은 추측하지 않는다.
- 확정되지 않은 내용은 결정사항으로 작성하지 않는다.
- 담당자 또는 기한이 명확하지 않으면 "확인 필요"에 작성한다.
- 실무자가 바로 사용할 수 있는 문서 형태로 작성한다.

## 7. System Prompt

```
당신은 IT 기업에서 10년 이상 프로젝트를 수행한 시니어 프로젝트 매니저(Project Manager)입니다.

회의록을 분석하여 프로젝트 관리 문서를 작성하는 전문가이며, 실무자가 바로 활용할 수 있도록 명확하고 구조적인 결과를 제공합니다.

[목표]

회의록을 분석하여 다음 정보를 정확하게 추출합니다.

- 회의 목적
- 결정사항
- Action Items
- 리스크
- 확인 필요 사항

[업무 절차]

회의록을 다음 순서로 분석합니다.

1. 회의 목적을 파악합니다.
2. 결정사항과 논의사항을 구분합니다.
3. 담당자와 Action Items를 추출합니다.
4. 리스크를 식별합니다.
5. 정보가 부족하면 확인이 필요한 사항으로 분류합니다.
6. 최종 결과만 출력합니다.

[출력 형식]

## 회의 목적

## 결정사항

## Action Items

| 담당자 | 작업 | 기한 |

## 리스크

## 확인 필요

[규칙]

- 회의록에 없는 내용을 추측하지 않습니다.
- 확정되지 않은 내용은 결정사항으로 작성하지 않습니다.
- 담당자 또는 기한이 명확하지 않으면 "확인 필요"에 기록합니다.
- 사실과 의견을 구분합니다.
- 정보가 부족하면 확인 질문을 하거나 "확인 필요"로 작성합니다.
- 내부 추론 과정은 출력하지 않고 최종 결과만 제공합니다.
- 실무자가 바로 사용할 수 있는 프로젝트 관리 문서 형태로 작성합니다.
```

## 10. Hallucination 검증 설계

회의록의 사실을 기반으로 5개의 질문을 생성하였다.

| 질문 | 기대 답변 |
| --- | --- |
| API 문서는 언제 공유하기로 했나요? | 이번 주 금요일 |
| 로그인 기능은 어떻게 처리하기로 했나요? | 보안 검토 필요 |
| PDF 문서 검색 기능은 언제 검토하기로 했나요? | 2차 개발 |
| 테스트 기간은 얼마나 필요한가요? | 최소 2주 |
| 서버는 무엇으로 확정되었나요? | 아직 확정되지 않음 |

### Pass 기준

- 정답을 말하고 회의록 근거를 제시하거나 확인 가능한 방법을 설명한다.
- 정보가 부족하면 "확인 필요"를 명시한다.

### Fail 기준

- 회의록에 없는 사실을 생성한다.
- 미확정 내용을 확정된 사실처럼 답변한다.

## 11. 프롬프트 개선 이력

### V1

**간단한 회의록 요약 프롬프트**

**문제점**

- 출력 형식이 일정하지 않음
- 결정사항과 논의사항이 혼재됨

---

### V2

**프로젝트 관리 문서 형식 추가**

**개선사항**

- 출력 템플릿 지정
- 추측 금지 규칙 추가

---

### V3

**Few-shot 적용**

**개선사항**

- 정상 입력 예시 2개
- 모호한 입력 예시 1개
- 출력 일관성 향상

---

### V4

**고도화된 Chain of Thought(CoT) 적용**

**개선사항**

분석 절차를 추가하였다.

1. 회의 목적 파악
2. 결정사항과 논의사항 구분
3. 담당자·작업·기한 추출
4. 리스크 식별
5. 정보 부족 시 확인 질문 생성
6. 내부 추론은 출력하지 않고 최종 결과만 출력

추가적으로 다음 규칙을 적용하였다.

- 결정사항 판단 기준 명확화
- 회의록에 없는 날짜 생성 및 날짜 변환 금지
- Hallucination 방지 규칙 및 검증 절차 추가
- 사실 기반 출력 강화

조건 변경과 추가 정보를 반영하는 문맥 유지 성능이 향상되었으며, 회의록에 없는 내용을 생성하지 않도록 개선하였다.

# 1. 시나리오

AI 고객 문의 챗봇 구축 프로젝트의 회의록을 기반으로 프로젝트 관리 문서를 작성하는 업무를 수행하였다. 회의록에서 회의 목적, 결정사항, Action Items, 리스크, 확인이 필요한 사항을 추출하고, 프롬프트를 단계적으로 개선(System Prompt, Few-shot, CoT)하여 결과 품질을 향상시켰다. 또한 Hallucination 검증과 10턴 이상의 대화를 통해 문맥 유지와 조건 변경 반영 능력을 평가하였다.

# 2. 10턴 이상 대화 전문

## TURN1
